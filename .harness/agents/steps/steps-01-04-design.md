## STEP 0 — Mode Detection (MANDATORY BEFORE ALL STEPS)

**Agent**: orchestrator (self)

Detect pipeline mode from `$ARGUMENTS`. **Explicit flags take priority over filesystem heuristics.**

```text
/myharness.orchestrator --N <path-to-spec-file>
/myharness.orchestrator --CR <feature-id> <path-to-cr-file>
/myharness.orchestrator <feature description>      # no flag → auto-detect
```

| Flag | Mode | Meaning |
|------|------|---------|
| `--N <spec-path>` | **CREATE** | New spec. Path to input spec file (e.g. `docs/input/new-spec/foo.md`). Orchestrator reads it to extract system name + modules. |
| `--CR <feature-id> <cr-path>` | **UPDATE** | Change request. Existing `feature-id` + path to CR file (e.g. `docs/input/change-request/cr-input.md`). Updates existing specs in-place. |
| *(none)* | auto | Auto-detect from filesystem (see below). |

`--N` and `--CR` are mutually exclusive.

**CASE A — `--N <spec-path>` present:**
- pipeline-mode = **CREATE**
- Read the spec file at `<spec-path>` as the SRS input.

**CASE B — `--CR <feature-id> <cr-path>` present:**
- pipeline-mode = **UPDATE** on `<feature-id>`
- Read the CR file at `<cr-path>`. Do NOT create a new branch/folder.

**CASE C — no flag (auto-detect):**

Scan `specs/` directory for an existing feature/module match before creating anything.

```
Run: Get-ChildItem specs/ -Directory | Select-Object -ExpandProperty Name
```

1. Extract module keyword from `$ARGUMENTS` (e.g., "MOD-01", "Objective ")
2. Look for matching folder in `specs/`
3. If match found:
   - Set `<feature-id>` = existing folder name
   - Set pipeline mode = **UPDATE** (do NOT create new branch/folder)
   - Propagate `<feature-id>` to ALL subsequent steps
4. If no match:
   - Set pipeline mode = **CREATE**

**Update Mode Rules:**

- **PROHIBITED:** Creating a new numbered folder when one already exists for this module.
- SRS/BD re-generation (Steps 1 & 2): Run normally — overwrite existing files.
- Spec update (Step 3): Invoke `myharness.specify` with: *"Update existing spec at `specs/<feature-id>/spec.md` in-place. DO NOT create new branch or folder."*

> Write `[STEP 0]` entry in orchestrator log per `protocols/log-formats.md`.

## STEP 1 — SRS Generation

> **PRE-CHECK (MANDATORY before dispatching):** Run this detection before deciding which SRS agent to invoke. There are **TWO** branches — evaluate in order.
>
> ```text
> Check if docs/output/srs-systems/srs-overview-system.md exists
> ```
>
> > 🧭 **No 1-vs-many module guessing.** The orchestrator does NOT count modules itself. `myharness.srs.system` is ALWAYS the SRS generator for a fresh run — it reads the spec, detects however many modules exist (1, 2, or N), and emits one overview + that many `modXX-*/` folders. A single-module spec simply yields one module folder. This removes the fragile "is it multi-module?" judgment entirely.
>
> ---
>
> > 🔁 **MODE GUARD (check pipeline-mode from STEP 0 FIRST):**
> > - **CREATE mode (`--N`)** → BRANCH A applies as written: an existing overview means it was already generated this run (or pre-supplied), so **skip regeneration**.
> > - **UPDATE mode (`--CR`)** → **do NOT skip.** The overview file exists from the original run, but the change request must be folded into the SRS. **Ignore BRANCH A and go to BRANCH B** — re-dispatch `myharness.srs.system` to **overwrite** `srs-systems/` with the CR applied (per the Update Mode Rule: *"SRS/BD re-generation — Run normally, overwrite existing files"*). Pass the CR file path to the agent as `change-request: <cr-path>` so it regenerates the affected modules. This matches the original single-SRS behavior, just at the system level.
>
> **BRANCH A — overview file EXISTS *and pipeline-mode = CREATE* (already generated this run, or pre-supplied):**
>
> 1. Set `srs-system.status: PRE_GENERATED` in `run-context.yaml`
> 2. Read `docs/output/srs-systems/srs-overview-system.md` — extract module list, FEA count, entity list
> 3. Populate `run-context.yaml` fields: `srs-system.module-count`, `srs-system.fea-count`, `srs-system.overview-path`, `srs-system.modules-dir = docs/output/srs-systems/`, and `steps.step-1-srs.path = docs/output/srs-systems/srs-overview-system.md`
> 4. Set `steps.step-1-srs.status: SKIPPED` with reason `pre-generated by myharness.srs.system`
> 5. Write `[STEP 1 SKIPPED — SRS pre-generated at docs/output/srs-systems/]` in orchestrator log
> 6. **DO NOT dispatch any SRS agent** — proceed directly to Step 1c.
>
> ---
>
> **BRANCH B — overview MISSING (CREATE), OR pipeline-mode = UPDATE (`--CR`) → dispatch `myharness.srs.system`:**
>
> No module counting. Just dispatch the system-SRS generator; it handles 1…N modules itself. Reached either because the overview is missing (fresh CREATE run) or because the MODE GUARD redirected an UPDATE/`--CR` run here to fold the change request into the SRS.
>
> 1. Set `srs-system.status: GENERATING` in `run-context.yaml`
> 2. **Dispatch `myharness.srs.system`** with the delegation block below. It writes `docs/output/srs-systems/srs-overview-system.md` + per-module folders `modXX-*/` (exactly as many modules as the spec contains — one folder for a single-module spec). In UPDATE mode it **overwrites** the existing `srs-systems/` files with the CR applied.
> 3. Parse its `<!-- STEP-RESULT -->` block (key `srs-system-path`). Verify `docs/output/srs-systems/srs-overview-system.md` now exists on disk.
> 4. Once written, set `PRE_GENERATED`, populate counts, mark `step-1-srs.status: SKIPPED` with reason `generated by myharness.srs.system` (or `regenerated from CR` in UPDATE mode). Proceed to Step 1c.
> 5. Write `[STEP 1 — system SRS generated via myharness.srs.system → N modules]` in orchestrator log (N = actual module count it reported).
>
> **`myharness.srs.system` delegation `$ARGUMENTS`:**
>
> ```yaml
> feature-id: <feature-id>
> input-spec: <the --N spec-path>              # CREATE mode
> change-request: <the --CR cr-path>           # UPDATE mode only — omit in CREATE mode
> mode-detail: <create | update>               # so the agent knows to overwrite vs create fresh
> output-dir: docs/output/srs-systems/
> pipeline-context: docs/output/run-logs/<feature-id>/run-context.yaml
> mode: autonomous
> language: English
> ```
>
> > ⚠️ **Single-owner rule:** `myharness.srs.system` is the SOLE writer of `docs/output/srs-systems/`. The orchestrator does NOT re-write or patch those files afterward — it only READS the STEP-RESULT block and updates `run-context.yaml`. (This eliminates the parent+child double-write seen in earlier runs.)
> >
> > **Hard-failure safety net:** If `myharness.srs.system` FAILS completely (no STEP-RESULT, or overview never written), log `[STEP 1 — srs.system FAILED]`, write an empty SRS stub note, and continue. Do NOT silently stop the pipeline.

> ➡️ **After STEP 1 completes, ALWAYS proceed to STEP 1b** (per-module SRS) — never skip straight to STEP 1c. STEP 1 produces the raw system-wide `srs-systems/`; STEP 1b turns each module into the formal per-module SRS that every downstream step consumes.

> ⛔ **[REPORT GATE]** per `protocols/report-gate-protocol.md`

## STEP 1b — Per-Module SRS (MANDATORY — runs once per module after STEP 1)

> **Purpose:** STEP 1 (`srs.system`) produced the canonical *system-wide* SRS under `docs/output/srs-systems/` (overview + one `modXX-*/` folder per module). STEP 1b runs `myharness.srs` **once per module** to materialize each module into the formal per-module SRS document `docs/output/design-docs/srs/srs-<MOD-ID>-<short-name>.md`. **This file is what STEP 2 (BD), STEP 3 (specify), STEP 5 (review), STEP 8 (DD) read.** If STEP 1b is skipped, those steps find no per-module SRS → pipeline breaks. **STEP 1b is NOT optional and NOT a fallback.**

> **Module loop:** Enumerate the modules by **listing the folders** under `srs-system.modules-dir` (`docs/output/srs-systems/`) matching `mod<XX>-*/` (this is the authoritative list; cross-check against `srs-system.module-count`). Dispatch `myharness.srs` for EVERY module folder found, passing that folder as `srs-module-path`. Per the *Parallel Execution Rules*, dispatch all modules **simultaneously** in one response, then barrier-wait for all before STEP 1c. For a single-module project this is just one call.

| Key | Value |
|-----|-------|
| Agent | `myharness.srs` |
| Runs | **once per module** (parallel) |
| Model | see catalog.yaml |
| Input | `srs-systems/srs-overview-system.md` + the module folder `srs-systems/mod<XX>-<slug>/` (detail + wireframe) |
| Output | `docs/output/design-docs/srs/srs-<MOD-ID>-<short-name>.md` (one per module) |
| Report | `reports/01b-srs-<mod-id>-report.md` |
| Gate | REPORT HARD GATE (per module) |
| On fail | Log error for that module, continue others with an empty SRS stub |

**Delegation `$ARGUMENTS` (one per module):**

```yaml
feature-id: <feature-id>
module-id: <mod-id>                  # e.g. MOD-01
module-keyword: <keyword>
srs-system-overview: docs/output/srs-systems/srs-overview-system.md
srs-module-path: docs/output/srs-systems/mod<XX>-<slug>/
pipeline-context: docs/output/run-logs/<feature-id>/run-context.yaml
mode: autonomous
language: English
```

> ⚠️ **Single-owner rule:** `myharness.srs` is the SOLE writer of its own `design-docs/srs/srs-<MOD-ID>-*.md`. The orchestrator does NOT re-open or re-patch it — only parse the STEP-RESULT and update `run-context.yaml`.

**After each module completes:** Parse the `<!-- STEP-RESULT -->` block and set, per module, `steps.step-1b-srs.<modXX>.path = docs/output/design-docs/srs/srs-<MOD-ID>-<short-name>.md` and `steps.step-1b-srs.<modXX>.status` in `run-context.yaml`. Downstream steps resolve the per-module SRS from this key.

> ⛔ **[REPORT GATE]** per `protocols/report-gate-protocol.md`

## STEP 1c — Artifact Compression (MANDATORY after Step 1b)

> Run immediately after Step 1b completes, before dispatching Step 2.
> Purpose: create lightweight summary files so Steps 2–7 read ~30% of the original token load.

| Key | Value |
|-----|-------|
| Agent | `myharness.compress` |
| Model | see catalog.yaml (lightweight tier — Haiku/GPT-5.4 mini) |
| Input | Per-module SRS path from `run-context.yaml steps.step-1b-srs.<modXX>.path` (the formal SRS produced by STEP 1b) |
| Output | `docs/output/design-docs/summaries/srs-<mod-id>-summary.md` |
| Report | `reports/01c-compress-report.md` |
| Gate | SOFT GATE — if compress fails, log warning and continue with full paths |

> Runs **once per module** (parallel), same as STEP 1b. Compress each module's `design-docs/srs/srs-<MOD>.md` into its own summary.

**Delegation `$ARGUMENTS` (one per module):**

```yaml
feature-id: <feature-id>
module-id: <mod-id>
srs-full-path: <from run-context steps.step-1b-srs.<modXX>.path>
bd-full-path: null   # BD not yet generated at this point
output-dir: docs/output/design-docs/summaries/
pipeline-context: docs/output/run-logs/<feature-id>/run-context.yaml
```

**After completion:** Update `run-context.yaml` summaries section:

```yaml
summaries:
  srs-summary-path: docs/output/design-docs/summaries/srs-<mod-id>-summary.md
  bd-summary-path: null   # populated after Step 2 + compress round 2
  srs-reduction-pct: <N>
```

**Downstream usage rule (orchestrator enforces):**
When passing SRS path to Steps 2, 3, 4, 5, 6, 7 — use `summaries.srs-summary-path` if it exists and is non-null. Pass the full path as `srs-full-path` alongside for agents that need to reference specific FR details.

> ⚠️ **SOFT GATE** — compress failure does NOT block the pipeline. Write `[WARN] Step 1c compress failed — falling back to full paths` in orchestrator log and continue normally.

## STEP 2 — BD Generation (External Design)

> **PRE-CHECK (MANDATORY before dispatching):** The per-module SRS from STEP 1b is the **primary** input.
>
> - **Primary SRS:** `steps.step-1b-srs.<modXX>.path` from `run-context.yaml` (the formal `design-docs/srs/srs-<MOD>.md` produced by STEP 1b). Pass as `srs-path`.
> - **System context:** also pass `srs-system-overview = srs-system.overview-path` so the BD agent can pull system NFR / ERD / global rules.
> - If `step-1b-srs.<modXX>.path` is missing (STEP 1b failed for this module), fall back to the module folder `srs-system.modules-dir/mod<XX>-<slug>/` raw detail.

| Key | Value |
|-----|-------|
| Agent | `myharness.bd` |
| Model | see catalog.yaml |
| Input | **Per-module SRS** `design-docs/srs/srs-<MOD-ID>-*.md` (primary, from STEP 1b), `docs/output/srs-systems/srs-overview-system.md` (system context), `docs/technical_architecture.md` |
| Output | `docs/output/design-docs/bd/bd-<MOD-ID>-<short-name>.md` |
| Report | `reports/02-bd-report.md` |
| Gate | REPORT HARD GATE + Auto-Resolve |
| On fail | Log error, continue with empty BD stub |

**Delegation `$ARGUMENTS`:**

```yaml
feature-id: <feature-id>
module-id: <mod-id>
module-keyword: <keyword>
pipeline-context: docs/output/run-logs/<feature-id>/run-context.yaml
srs-path: <from run-context steps.step-1b-srs.<modXX>.path>           # primary per-module SRS
srs-system-overview: docs/output/srs-systems/srs-overview-system.md   # system NFR/ERD context
auto-resolve: true
```

**After completion:** Parse the `<!-- STEP-RESULT -->` block and update `run-context.yaml`. Do **NOT** re-open or re-write the BD file.

> > ⚠️ **Single-owner rule (prevents the parent+child double-write):** `myharness.bd` is the SOLE writer of `docs/output/design-docs/bd/bd-<MOD-ID>-*.md` and its report. The orchestrator must **NOT** re-open, re-patch, or re-save the BD file after the agent returns — it only READS the STEP-RESULT block and updates `run-context.yaml`.
> >
> > **Auto-resolve happens INSIDE the BD agent, before it saves** — the BD agent resolves all `[NEEDS CLARIFICATION]` markers with optimal assumptions (per `protocols/auto-resolve-protocol.md`) and writes them into the BD body in the SAME pass, so no second write is needed. Pass `auto-resolve: true` in the delegation `$ARGUMENTS` so the agent knows to resolve-then-save in one shot. If, after the agent returns, unresolved markers remain, log a `[WARN]` and continue — do NOT re-edit the file to fix them.

> ⛔ **[REPORT GATE]** per `protocols/report-gate-protocol.md`

## STEP 4 — Consolidated Spec Clarification (Autonomous Mode)

| Key | Value |
|-----|-------|
| Agent | `myharness.clarify` |
| Model | see catalog.yaml |
| Input | `specs/<feature-id>/spec.md` |
| Output | Updated spec + `reports/04-clarify-qa.md` |
| Report | `reports/04-clarify-report.md` |
| Gate | REPORT HARD GATE (+ QA Summary section required) |

**Autonomous behavior (NO PAUSE):**

1. `myharness.clarify` identifies ambiguities → produces QA list
2. orchestrator applies Auto-Resolve Protocol to every question
3. orchestrator encodes all answers back into spec
4. orchestrator writes QA list with answers to `reports/04-clarify-qa.md`
5. Confirms no `[NEEDS CLARIFICATION]` markers remain

**Output format for `04-clarify-qa.md`:**

```markdown
# Clarification Q&A — Auto-Resolved

| # | ID | Question | Auto-Answer | Rationale | Confidence |
|---|----|----------|-------------|-----------|------------|
| 1 | TBC-01 | ... | ... | ... | High |

## Summary
- Total questions: N
- Auto-resolved: N (High: X, Med: Y, Low: Z)
- Pending user confirmation: 0 (pipeline continues)
```
